# <a name="data-management-in-retail-overview"></a>用于零售业的数据管理概述

## <a name="introduction"></a>介绍

数据是开发和提供更好的零售体验的基础。 数据存在于零售组织的各个方面，可用于从整个价值链中提取见解并应用到运营绩效和客户行为中，还可用于支持改进服务体验。 从联机浏览到社交活动再到店内购买，数据俯拾皆是。 但是，捕获数据只是数据管理的一部分。 将不同的数据拼接在一起进行分析需要在整个组织内正确处理数据，从而使零售商更有能力在运营业务时做出有影响力的决策。

例如，随着移动购物的增长，客户已经开始期望零售商具有关于其购物习惯的合理数量的数据以用于改善体验。 一个用例示例是，当客户在实体零售店内的特定位置购物时直接向客户的移动设备发送个性化产品和促销产品。 利用有关内容、地点、方式、数量和频率的数据，以及商店产品可用性等附加输入，可在客户在目标产品附近购物时向客户设备发送实时促销信息。 

有效的数据使用可帮助零售商提供更相关的体验从而推动客户进行购买，例如，零售商可能会向客户发送包含零售商电子商务网站折扣代码的通知。  此外，这些数据将提供可操作的见解，使公司领导者可基于数据做出决策并指导他们的行动

提供促销信息的操作由数据点组合通知，并在顾客进入商店时触发。 进行这些连接并生成相应操作的能力基于下面显示的数据管理模型。

将数据导入 Azure 时，请考虑数据源的 3P 及其对零售商希望启用的方案的适用性。 数据源的 3P 即购买、公用和专用。

> 购买的数据通常会增加和增强组织的现有数据，往往是补充组织数据捕获范围的市场和人口统计数据。 例如，零售商可以购买额外的人口统计数据以增加主客户记录，从而确保记录完整准确。 
>
> 公用数据是免费提供的，可以从社交媒体、政府资源（例如地域）和其他联机资源中获取。 该数据可以推断诸如与购买模式相关的天气模式或在特定地域中表示产品流行度的社交活动等见解。 公共数据通常通过 API 提供。
>
> 专用数据是保留在组织内的数据。 它可能是零售商的本地系统、SaaS 应用程序或云提供商。 要访问 SaaS 应用程序提供程序中的数据以及其他供应商数据，API 通常用于与供应商的系统进行通信。 这包括电子商务站点日志、POS 销售数据和库存管理系统等数据。

这些不同的数据类型用于来自数据管理管道的各种见解。

## <a name="ingest"></a>引入

最初，数据以其本机格式加载到 Azure 中，并相应地存储。 接收和管理不同的数据源可能令人生畏，但 Microsoft Azure 提供的服务可以快速、轻松地将数据加载到云中，使其可用于数据管理管道中的处理过程。 

Azure 提供了一些有用的迁移数据服务。 具体选择哪种服务取决于要迁移的数据类型。 适用于 SQL Server 的 [Azure 数据迁移](/azure/dms/dms-overview?WT.mc_id=retaildm-docs-dastarr) 服务和 [Azure 导入/导出服务](/azure/storage/common/storage-import-export-service?WT.mc_id=retaildm-docs-dastarr)有助于将数据导入 Azure。 要考虑的其他数据入口服务包括 [Azure 数据工厂](/azure/data-factory?WT.mc_id=retaildm-docs-dastarr)和 [Azure 逻辑应用](/azure/logic-apps/?WT.mc_id=retaildm-docs-dastarr)连接器。 每种服务都有自己的功能，应调查研究，看看哪种技术最适合给定情况。

数据引入并不仅限于 Microsoft 技术。 通过 [Azure 市场](https://azuremarketplace.microsoft.com?WT.mc_id=retaildm-docs-dastarr)，零售商可以在 Azure 中配置许多不同的供应商数据库，以便与现有本地系统配合使用。 

并非所有数据都必须在 Azure 中维护。 例如，销售点 (POS) 数据可以在本地保存，因此 Internet 中断不会影响销售交易。 该数据可以按计划（可能是每晚或每周）排入队列并上传到 Azure，以供分析使用，但始终将本地数据视为事实来源。

## <a name="prepare"></a>准备

在开始分析之前，必须准备好数据。 这种数据构形对于确保预测模型的质量、报告 KPI 和数据的相关性非常重要。

在准备用于分析的数据时，有两种类型的数据需要解决，即结构化数据和非结构化数据。 结构化数据更容易处理，因为它已经形成并已格式化。 结构化数据可能只需一个简单的转换就可以从源格式转变为可以用于分析格式。 非结构化数据通常会带来更多挑战。 非结构化数据不以固定记录长度格式存储。 示例包括文档、社交媒体源以及数字图像和视频。 这些数据的管理方式必须与结构化数据不同，并且通常需要专门的流程来确保这些数据以可用的方式存储在正确的数据存储中。

数据构形发生在准备阶段的提取、转换和加载 (ETL) 过程中。 从导入 Azure 的未更改数据源中提取数据，根据需要进行“清除”或重格式化，并以新的、更结构化的格式存储。 常见的 ETL 数据准备操作是将 .csv 或 Excel 文件转换为 parquet 文件，方便 Apache Spark 等机器学习系统更快速读取和处理。 另一种常见方案是从 .csv 文件或其他格式创建 XML 文件或 JSON。 生成的格式更易于与其他分析引擎一起使用。

在 Azure 中，有几种转换技术可用作重构数据的 ETL 服务。 选项包括 [Azure Databricks](/azure/azure-databricks?WT.mc_id=retaildm-docs-dastarr)、[Azure Functions](/azure/azure-functions/?WT.mc_id=retaildm-docs-dastarr) 或逻辑应用。 Databricks 是 Apache Spark 的完全托管实例，用于将数据从一种形式转换为另一种形式。 Azure Functions 是无状态（或“无服务器”）函数，具有可触发自身并运行代码的触发器。 逻辑应用集成了服务。

## <a name="store"></a>存储

在处理前存储数据需要慎重考虑。 数据可以采用结构化或非结构化格式，数据的构形通常决定其存储目标。 例如，高度结构化的数据可能适用于 Azure SQL。 较低结构化的数据可以保存在 blob 存储、文件存储或表存储中。

存储在 Azure 中的数据具有出色的性能，有可靠的服务级别协议 (SLA) 支持。 数据服务提供了更易于管理的解决方案、高可用性、跨多个地理位置的复制，并且最重要的是，Azure 提供驱动机器学习所需的数据存储和服务。

结构化和非结构化数据都可以存储在 [Azure Data Lake](/azure/data-lake-store/data-lake-store-overview?WT.mc_id=retaildm-docs-dastarr) 中，并使用 [U-SQL](/azure/data-lake-analytics/data-lake-analytics-u-sql-get-started?WT.mc_id=retaildm-docs-dastarr)（一种特定于 Azure Data Lake 的查询语言）进行查询。 可以包括在 Data Lake 中的数据示例包括以下内容，分为常规结构化和非结构化数据源。

### <a name="structured-data"></a>结构化数据

- CRM 数据和其他业务线应用程序
- POS 交易数据
- 传感器数据
- 关系数据
- 电子商务交易数据

### <a name="unstructured-data"></a>非结构化的数据

- 社交源
- 视频
- 数字图像
- 网站点击流分析

支持非结构化数据以产生价值的用例越来越多。 这是由数据驱动决策的需求和 AI 等技术的进步推动的，从而可以大规模捕获和处理数据。 例如，数据可以包括照片或流媒体视频。 例如，可以利用流媒体视频来检测客户购物选择以进行无缝结账，或者产品目录数据可以与顾客最喜欢的衣服的照片无缝合并，以提供类似或推荐物品的视图。 

结构化数据的示例包括关系数据库数据馈送、传感器数据、Apache Parquet 文件和电子商务数据。 这些数据的固有结构使它们非常适合机器学习管道。

Azure Data Lake 服务还可以使用 [Data Lake Analytics](/azure/data-lake-analytics/data-lake-analytics-overview?WT.mc_id=retaildm-docs-dastarr) 启用批处理和交互式查询以及实时分析。 此外，Data Lake 特别适用于非常大的数据分析工作负荷。 最后，Data Lake 中的数据将持续存在，没有时间限制。

其他数据存储（如关系数据库、Blob 存储、Azure 文件存储和 Cosmos DB 文档存储）也可以保存纯净数据，以便在数据管理管道中进行下游分析。 不存在使用 Data Lake 的要求。

## <a name="analyze"></a>分析

对于降低库存成本等问题，零售商可以使用机器学习进程执行的分析。

数据分析通过机器学习引擎准备要处理的数据，从而获取客户体验的深层次见解。 该过程会产生可以“学习”的模型，并可应用于未来数据以预测结果。 模型定义将要检查的数据以及如何通过各种算法分析数据。 使用分析的输出数据和数据可视化效果是可以触发见解的操作，例如利用零售商电子商务平台中的客户愿望清单提供商品的店内优惠券。

通过向学习生态系统提供用于处理的存储数据来进行数据分析。 通常情况下，这是由 Hadoop、Databricks 或在虚拟机上运行的自托管 Spark 实例执行的机器学习。 这也可以通过查询数据来完成。 通常可以在纯净数据中找到针对 KPI 的见解，而无需通过机器学习管道。

[Hadoop](/azure/hdinsight/hdinsight-hadoop-architecture?WT.mc_id=retaildm-docs-dastarr) 是完全托管的 Azure 服务 [HDInsight](/azure/hdinsight/?WT.mc_id=retaildm-docs-dastarr) 的一部分。 HDInsight 是一组数据学习工具，用于定型数据模型、将数据输出到数据仓库以及通过 Hive 查询语言对 Hadoop 执行查询。 HDInsight 可以分析流媒体数据或历史数据。

各种学习算法可作为定型的一部分应用于数据，并可维护数据模型。 数据模型显式确定为分析人员生成的数据结构。

首先，清理数据并正确形成数据。 然后由机器学习系统（如 HDInsight 或 Apache Spark）进行数据处理。 为此，现有数据用于为模型定型，而模型又用于数据分析。 定型后的模型将定期更新，纳入新的已知良好数据，以提高分析的准确性。 机器学习服务使用该模型对正在处理的数据执行分析。

在模型定型和运行数据分析过程之后，从机器学习分析派生的数据可以存储在数据仓库或标准化存储数据库中用作分析数据。 Microsoft 提供了 [Power BI](/power-bi/?WT.mc_id=retaildm-docs-dastarr)，这是一个功能齐全的数据分析工具，用于深入分析数据仓库中的数据。

## <a name="action"></a>操作

零售业中的数据不断变化，处理此数据的系统必须及时改变。 例如，需要快速处理电子商务购物者数据。 这样，买家购物车中的商品可用于在结帐过程中提供附加服务或附加商品。 这种形式的数据处理和分析几乎必须立即进行，并且通常由执行“微批次”交易的系统执行。 也就是说，在可以访问已处理数据并通过模型运行的系统中分析数据。

其他“批处理”操作可以定期发生，但不需要近乎实时地发生。 当批量分析在本地进行时，这些作业通常在晚上、周末或资源未占用时运行。 使用 Azure，可以随时缩放所需的大型批处理作业和虚拟机来支持它们。

从以下步骤开始。

1. 为数据存储创建数据引入计划，为要执行的分析提供值。 详细的数据同步或迁移计划就绪后，可以将数据以原始格式导入 Azure。 

2. 确定所需的可操作见解，并选择数据处理管道以适应数据处理活动。 

3. 考虑到这些数据特征，使用适当的算法创建数据处理管道，以获得所需的见解。

4. 如果可能，将通用数据模型用于输出到数据仓库，这可以揭示最有趣的数据功能。 这通常意味着读取原始 Azure 存储系统中的数据并将清理后的版本写入其他数据存储。

5. 通过 Spark 或 Hadoop 提供的机器学习管道处理数据。 然后将输出馈送到数据仓库。 有许多默认的数据处理算法，或零售商可以实现自己的算法。 除了 ML 方案外，还要将数据加载到标准数据存储中并强制执行通用数据模型，然后查询 KPI 数据。 例如，数据可以存储在星型架构或其他数据存储中。

随着数据可供数据分析师使用，可能会发现可操作的见解，并可将这些新知识应用到实践中。 例如，客户的购买偏好可以被加载回零售商的系统中，并用于改进若干客户接触点（如下所示）。

- 通过捆绑产品来增加电子商务或 POS 平均交易量
- 购买 CRM 中的历史记录以支持客户呼叫中心查询
- 由电子商务推荐引擎定制的产品建议
- 基于客户数据的定向广告和相关广告
- 根据供应链中的产品移动更新库存可用情况

可能出现的另一种见解是先前未被质疑的模式。 例如，可能会发现在下午 3:00 到下午 7:00 之间出现更多库存损失。 这可能意味着需要额外的数据来确定根本原因和操作方案，例如改进的安全性或标准操作程序。

用于零售业的数据管理非常复杂。 但它提供了交付相关性和改进客户体验的宝贵能力。 使用本文中的技术可以获得有益的见解，推动改进客户体验，实现可盈利的业务成果，并发现推动业务进步的趋势。
